Authorization

Authorization is the function of specifying access rights to resources and functionality made available by the TIDAL API and the TIDAL SDK, i.e. granting applications or users access to the resources they serve. TIDAL Developer Platform implements OAuth 2.1, an industry standard protocol for authorization.

In order for an application to successfully access functionality through the TIDAL API or SDK, it needs to supply credentials in the form of a valid access token. An access token is a credential in the form of a string, representing an authorization issued to the client.

TIDAL Developer Platform supplies three ways, or "flows", for obtaining access tokens - Client credentials, Authorization code and Refresh token - each outlined in detail in the corresponding section below.

NOTE: It is recommended to rely on the Auth module in the TIDAL SDK for obtaining access tokens whenever possible. Relying on the Auth module will not only simplify authorization handling in your application, it will also guarantee full compatibility with the TIDAL Developer Platform - our own applications rely on it - and will allow you to easily integrate other modules from the TIDAL SDK into your application.

  • Client credentials flow - This flow allows applications to get access tokens outside of the context of a user, meaning these tokens can only be used to access resources not owned by any user.

  • Authorization code flow - Use this flow to allow TIDAL users to login through your application, granting your application access to their resources.

  • Refresh token flow - The refresh token flow allows you to easily get a new access token when the old one has expired.

If you choose not to build using our SDK, and you already have a standard OAuth implementation, you can find our standard OAuth protocol endpoints at:

  • Authorization endpoint: https://login.tidal.com/authorize

  • Token endpoint: https://auth.tidal.com/v1/oauth2/token

Access tokens

An access token is a credential in the form of a string, representing an authorization issued to the client. Access tokens are used to access protected resources in the TIDAL API.

Each request towards the TIDAL API must supply a valid access token in the HTTP header in order to succeed: 

"Authorization: Bearer <ACCESS_TOKEN>"

The following example shows how to make a request towards the TIDAL API using curl, correctly supplying the access token in the Authorization header:

curl -X "GET" \
  -H "accept: application/vnd.tidal.v1+json" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "Content-Type: application/vnd.tidal.v1+json" \
  "https://openapi.tidal.com/v2/albums/59727856?countryCode=US"

Scopes

Scope is a mechanism in OAuth 2 to limit an application's access to a user's account. An application can request one or more scopes, this information is then presented to the user in the consent screen, and the access token issued to the application will be limited to the scopes granted.

TIDAL Developer Platform utilises scopes to guarantee that TIDAL users always remain in control of what they share with third-party applications.

Visit the TIDAL API reference for up-to-date documentation of which endpoints require which scopes.

Client credentials flow

The client credentials flow can be used when the authorization scope is limited to the protected resources under the control of the client. For the TIDAL Developer Platform, this means that access tokens obtained using this flow can be used to access resources that are not owned by any specific user, for example the TIDAL Catalog.

It is recommended to rely on the TIDAL SDK for handling authorization in your application whenever possible. If you cannot, or do not want to use the SDK, the Web API for this flow is also documented below.

Web API

In order to obtain an access token using the client credentials flow, execute the following two commands in your terminal to make the correct call to the TIDAL token endpoint. Make sure to substitute <CLIENT_ID> and <CLIENT_SECRET> with the corresponding credentials for your app.

B64CREDS=$(echo -n "<CLIENT_ID>:<CLIENT_SECRET>" | base64)
curl -X POST \
  -H "Authorization: Basic $B64CREDS" \
  -d "grant_type=client_credentials" \
  "https://auth.tidal.com/v1/oauth2/token"

If all goes well, you should receive a response containing the access token similar to this one:

{
  "access_token": "<ACCESS_TOKEN>",
  "token_type": "Bearer",
  "expires_in": 86400
}

Authorization code flow

The authorization code flow is used to obtain both access tokens and refresh tokens, and allows TIDAL users to log in through your application, granting your application access to their resources.

There are two key steps involved when obtaining access (and refresh) tokens using this flow. First - your application needs to obtain authorization from the end user (in the form of an authorization code), and second - your application need to exchange the authorization code for an access token and and refresh token.

It is recommended to rely on the TIDAL SDK for handling authorization in your application whenever possible. If you cannot, or do not want to use the SDK, the Web API for this flow is also documented below.

SDK

Web API

Follow the directions below to obtain an access token and refresh token using the authorization code flow via the Web API.

Generate code verifier and code challenge

The authorization code flow is susceptible to authorization code interception attacks. In order to mitigate this threat, TIDAL (and OAuth 2.1) enforces Proof Key for Code Exchange (PKCE) for all clients. PKCE defines and utilizes a dynamically created cryptographically random key called "code verifier" and its transformed value, called "code challenge". See the PKCE specification and the S256 PKCE code challenge appendix for more information on how to generate the code verifier and code challenge.

These values are referred to as <CODE_VERIFIER> and <CODE_CHALLENGE> below.

Obtain authorization from end user

In order for your application to obtain authorization from the end user, you will need to present the TIDAL Login Service authorization dialogue to the user by making a GET request to the authorize endpoint using the following URL:

https://login.tidal.com/authorize
  ?response_type=code
  &client_id=<CLIENT_ID>
  &redirect_uri=<REDIRECT_URI>
  &scope=<SCOPES>
  &code_challenge_method=S256
  &code_challenge=<CODE_CHALLENGE>
  &state=<STATE>

  • <CLIENT_ID> - The client id of your app. See Manage apps.

  • <REDIRECT_URI> - The redirect URI you have configured for your app. See Manage apps.

  • <SCOPES> - The list of scopes that you want your application to obtain.

  • <CODE_CHALLENGE> - The S256 PKCE code challenge generated above.

  • <STATE> - An optional parameter that provides protection against attacks such as cross-site request forgery.

Upon success, the TIDAL Login Service will redirect to the following address:

<REDIRECT_URI>
  ?code=<CODE>
  &state=<STATE>

  • <REDIRECT_URI> - The redirect_uri parameter specified in the call to authorize.

  • <CODE> - The authorization code to be exchanged for an access token.

  • <STATE> - Loopback of the optional state parameter sent to the TIDAL Login Service.

In case an error occurred, or if the user did not consent to your application accessing their data, the TIDAL Login Service will redirect to the following address:

<REDIRECT_URI>
  ?error=<ERROR>
  &error_description=<ERROR_DESCRIPTION>
  &state=<STATE>

  • <REDIRECT_URI> - The redirect_uri parameter specified in the call to authorize.

  • <ERROR> - Error code describing the reason the authorization failed.

  • <ERROR_DESCRIPTION> - Description of why the authorization failed.

  • <STATE> - Loopback of the optional state parameter sent to the TIDAL Login Service.

Exchange authorization code for token

Once your application has successfully obtained an authorization code, it needs to exchange it for an access and refresh token by calling the TIDAL token endpoint:

curl -X POST \
  -d "grant_type=authorization_code" \
  -d "client_id=<CLIENT_ID>" \
  -d "code=<CODE>" \
  -d "redirect_uri=<REDIRECT_URI>" \
  -d "code_verifier=<CODE_VERIFIER>" \
  "https://auth.tidal.com/v1/oauth2/token"

  • <CLIENT_ID> - The client id of your app. See Manage apps.

  • <CODE> - The authorization code received in the previous step.

  • <REDIRECT_URI> - The redirect_uri parameter specified in the previous call to authorize. Used for validation only.

  • <CODE_VERIFIER> - The code verifier used when you generated the <CODE_CHALLENGE>.

If all goes well, you should receive a response containing the access token similar to this one:

{
  "access_token": "<ACCESS_TOKEN>",
  "token_type": "Bearer",
  "scope": <SCOPES_OF_ACCESS_TOKEN>,
  "expires_in": 86400,
  "refresh_token": "<REFRESH_TOKEN>"
}

Refresh token flow

Refresh tokens are credentials used to obtain access tokens, and the refresh token flow allows your application to easily get a new access token when the current one has expired.

It is recommended to rely on the TIDAL SDK for handling authorization in your application whenever possible. If you cannot, or do not want to use the SDK, the Web API for this flow is also documented below.

SDK

When using the Auth module from the TIDAL SDK, the access token will automatically get refreshed if needed when your application calls the CredentialsProvider.getCredentials function.

Web API

Make the following call to the TIDAL token endpoint in order to obtain an access token using a refresh token.

curl -X POST \
  -d "grant_type=refresh_token" \
  -d "refresh_token=<REFRESH_TOKEN>" \
  "https://auth.tidal.com/v1/oauth2/token"

If all goes well, you should receive a response containing the access token similar to this one:

{
  "access_token": "<ACCESS_TOKEN>",
  "token_type": "Bearer",
  "expires_in": 86400,
  "scope": <SCOPES_OF_ACCESS_TOKEN>
}
